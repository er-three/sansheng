## 系统性缺口

### 1. 语言不一致
```
中文：皇帝、中书省、门下省、尚书省
英文：吏部、工部、户部、兵部、刑部、库部
```
混用不影响功能，但增加维护认知成本。

### 2. 输出格式标准不统一
```
中书省：JSON ✅
工部：JSON ✅
门下省：自由文本 ❌
尚书省：自由文本 ❌
```
Agent 之间传递结果靠自由文本，一旦上游改变表述，下游解析就可能出错。

### 3. Agent 间没有共享的输入输出契约
每个 Agent 只知道自己的 I/O，没有一个中央文档描述"中书省的输出 = 尚书省的输入格式"。当 schema 变化时，要手动更新多个文件。

### 4. 工具声明与实际使用不一致

| Agent | 使用的工具 | permission 里声明了吗 |
|-------|-----------|-------------------|
| 门下省 | read（检查输入文件） | ❌ 没有 |
| 户部 | read/glob/grep（补充上下文） | ❌ 没有 |
| 刑部 | semantic_grep | ❌ 没有 |

## 优化优先级

| 优先级 | 改动 | 原因 |
|--------|------|------|
| 🔴 高 | 门下省拆分 + 权限修复 | 影响核心流程正确性 |
| 🔴 高 | 中书省加 read 权限 | 隐性依赖会导致规划出错 |
| 🟡 中 | 统一输出格式为 JSON | 提升 Agent 间传递的可靠性 |
| 🟡 中 | 新增 Agent 间契约文档 | 防止 schema 漂移 |
| 🟢 低 | 语言统一 | 认知成本，不影响功能 |
| 🟢 低 | 礼部边界澄清 | 轻微重叠，暂时不影响运行 |

---

## 结论

当前提示词能跑，但不够健壮。核心问题是权限声明和实际行为不匹配，以及缺少 Agent 间的结构化契约。建议先修完边界问题（见 BOUNDARY_OPTIMIZATION_DESIGN.md），再做输出格式统一。
