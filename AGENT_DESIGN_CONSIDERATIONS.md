# Agent 设计通用注意事项

**分析日期**: 2026-03-16
**范围**: 通用 Agent 设计，不限于本项目

---

## 1. 上下文污染（Context Contamination）

```
Agent A 执行完把大量中间数据塞进 prompt
Agent B 收到时上下文已经很长
Agent C 开始"忘记"早期的指令
```

**本项目风险**：尚书省 steps: 100，长流程下早期的执行原则可能被稀释。

**解法**：每次 task 调用只传该步骤需要的最小信息，不传历史。

---

## 2. 幂等性（Idempotency）

Agent 重试时，同一操作执行两次会不会产生副作用？

```
工部第一次写文件成功，但验收超时
尚书省重试，工部再写一次 → 文件内容被覆盖/重复
```

**本项目**：工部有"创建前检查是否存在"，部分覆盖，但不完整。

---

## 3. 确定性 vs 创造性的 temperature 选择

```
当前：
皇帝 temperature: 0.2  （战略决策，应该偏低）
中书省 temperature: 0.1 （规划，合理）
工部 temperature: 0.1   （代码实现，合理）
```

**问题**：皇帝做最终验收时需要判断"业务价值是否达成"，这是需要一点创造性思维的，0.2 偏低。

**建议**：验收场景用 0.3~0.5。

---

## 4. Agent 的"越权"风险

模型会自作聪明。即使 prompt 写了"不做判断"，在某些情况下 Agent 仍会：
- 自行修改计划（应该是中书省的事）
- 自行决定跳过某步（应该上报）
- 用自然语言"解释"失败而不是结构化报告

**解法**：在 prompt 末尾加负向约束，明确列出禁止行为，比"只做 X"更有效。

---

## 5. 死锁场景

```
尚书省等门下省验收
门下省等吏部提供文件信息
吏部在等工部写文件
工部在等尚书省下达指令
```

循环依赖在 Agent 系统里比代码里更难发现，因为没有编译器报错。

**解法**：每个 Agent 设置超时上报机制，等待超过 N 步没有响应就主动上报皇帝。

---

## 6. 模型能力与任务信息匹配

```
当前：
皇帝 → Opus（战略）✅
刑部 → Opus（代码审查）✅
尚书省 → Haiku（调度）✅
吏部 → Haiku（读文件）✅
```

**风险点**：中书省用 Opus 规划，但规划质量高度依赖它对 domain.yaml 的理解。如果读不到文件（当前权限问题），Opus 的推理能力再强也没用——能力和信息不匹配。

---

## 7. 没有 Dry Run 机制

真实 Agent 系统里，能不能在不执行的情况下预演一遍流程，是健壮性的重要指标。

```
用户："帮我跑逆向工程"
理想：先 dry run，展示将要做什么，用户确认后再执行
现状：直接执行，出错了才知道哪步有问题
```

---

## 8. 观测性（Observability）

Agent 执行过程是黑盒，出问题时：
- 哪个 Agent 做了什么决策？
- 为什么选了这个分支？
- 哪步开始偏离？

**本项目**：只有 pipeline_status，没有结构化的执行日志。

**解法**：每个 Agent 完成后输出标准化的 execution_log，集中存到一个文件里。

---

## 总结

| 维度 | 当前覆盖 | 风险等级 |
|------|---------|---------|
| 上下文污染 | 部分（最小输入原则） | 🟡 中 |
| 幂等性 | 部分（文件存在检查） | 🟡 中 |
| Temperature 匹配 | 基本合理 | 🟢 低 |
| 越权风险 | 有负向约束但不完整 | 🟡 中 |
| 死锁场景 | 未覆盖 | 🔴 高 |
| 模型能力与信息匹配 | 有缺口 | 🔴 高 |
| Dry Run | 未覆盖 | 🟡 中 |
| 观测性 | 弱 | 🟡 中 |

**最值得优先处理的两个**：死锁场景和观测性——这两个问题在系统跑起来之前看不出来，但一旦出问题排查成本极高。
