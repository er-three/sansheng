## 📊 完整工作流

### 第0阶段：目标设定（皇帝）

```
皇帝设定目标
  ├─ 目标描述（自然语言）
  ├─ Domain选择（domain.yaml）
  ├─ 时间和资源约束
  └─ 业务验收标准

例如：
  "重构支付模块，提高性能，确保向后兼容"
  Domain: "payment-refactor"
  时间: 1周
  标准: 性能提升20%，0数据丢失
```

**→ 进入第1阶段**

---

### 第1阶段：规划（中书省）

#### 输入
```json
{
  "objective": "用户的目标描述",
  "domain_yaml": "参考domain.yaml",
  "constraints": {
    "time": "1周",
    "resources": "3人",
    "budget": "..."
  }
}
```

#### 中书省的工作
```
1. 理解需求
   ├─ 阅读domain.yaml的pipeline定义
   ├─ 理解各个skill的用途
   └─ 确认dependencies

2. 分析现状
   ├─ 查看现有代码结构
   ├─ 理解关键模块的交互
   └─ 识别风险点

3. 设计方案
   ├─ 选择实现路线
   ├─ 分析备选方案
   └─ 确定最优路径

4. 拆解步骤
   ├─ 识别各个Step
   ├─ 分析Step间的依赖关系
   ├─ 为每个Step分配六部
   └─ 定义验收标准

5. 生成计划
   ├─ plan.json (所有Step详情)
   ├─ dependencies.json (依赖关系图)
   ├─ acceptance_criteria.json (验收标准)
   └─ summary.md (计划说明)
```

#### 输出
```json
{
  "plan": {
    "domain": "payment-refactor",
    "steps": [
      {
        "id": "step-1",
        "name": "分析现有支付流程",
        "uses": ["yibu"],
        "dependencies": [],
        "input": {
          "source_dir": "src/payment/",
          "rules": ".eslintrc"
        },
        "acceptance_criteria": {
          "must_cover": "所有payment相关代码",
          "report_issues": true
        }
      },
      // ... more steps
    ],
    "dependency_graph": {...},
    "critical_path": ["step-1", "step-2", "step-4"]
  }
}
```

**→ 进入第2阶段**

---

### 第2阶段：审核（门下省Phase 1）

#### 输入
中书省的plan.json

#### 门下省的工作
```
审核checklist:
  ☐ 所有Step都有明确的uses字段
  ☐ 没有循环依赖
  ☐ 分配的六部都合理
  ☐ 验收标准都可衡量
  ☐ 关键路径清晰
  ☐ 工期估计合理
```

#### 决策
```
IF [审核通过]:
  → APPROVED, 继续第3阶段
ELSE:
  → REJECTED, 返回中书省修改
     └─ 中书省修改后重新提交
```

**→ 进入第3阶段**

---

### 第3阶段：执行调度（尚书省）

#### 执行循环

```
初始化：
  completed_steps = []
  in_progress_steps = []

WHILE (还有Step未执行):

  1. 找出所有可执行Step
     ├─ dependencies都完成
     └─ 可以并行执行多个

  2. 并行调用六部
     ├─ for each step in ready_steps:
     │    task(agent="ministry", prompt="...")
     └─ 等待所有task完成

  3. 收集输出
     ├─ 每个六部返回执行结果
     └─ 记录时间戳和日志

  4. 执行层验证（轻量级）
     ├─ 检查JSON格式
     ├─ 检查null值
     └─ 记录任何问题

  5. 报告给门下省
     └─ 详细的Step执行结果

  6. 等待门下省决策
     └─ 继续 / 重做 / 上报皇帝

  7. 执行门下省的指示
     └─ 更新in_progress_steps
```

#### 并行执行示例

```
假设计划中有：
  Step 1: 分析(yibu) [dependencies: none]
  Step 2: 获取文档(hubu) [dependencies: none]
  Step 3: 实现代码(gongbu) [dependencies: step-1, step-2]
  Step 4: 测试(bingbu) [dependencies: step-3]

执行顺序：
  时刻T0:
    ├─ 并行: task(yibu, ...) + task(hubu, ...)
    └─ 等待

  时刻T1 (两个都完成):
    ├─ 并行: task(gongbu, ...)
    └─ 等待

  时刻T2 (gongbu完成):
    ├─ 并行: task(bingbu, ...)
    └─ 等待

  时刻T3 (bingbu完成):
    └─ 所有Step完成，进入验证阶段

效率：
  串行执行: T0-T1 + T1-T2 + T2-T3 = 总耗时
  并行执行: max(T1-T2, T2-T3) + T1 = 更快!
```

**→ 进入第4阶段**

---

### 第4阶段：逐步验证（门下省Phase 2）

#### 为每个Step进行验证

```
尚书省报告："Step 1完成"
  ↓
门下省验证：
  ├─ 检查输出是否存在
  ├─ 验证输出格式是否正确
  ├─ 检查是否符合acceptance_criteria
  └─ 决策:

    ✅ 符合 → PASS
       ├─ 记录通过
       └─ 指示尚书省："继续下一步"

    ❌ 不符合 → [第几次失败?]
       ├─ 第1次:
       │   ├─ 分析失败原因
       │   ├─ 可以调整标准吗？
       │   └─ 指示尚书省："让[ministry]重做"
       │
       └─ 第2次:
           ├─ 记录两次失败的细节
           └─ 上报皇帝："需要人工决策"
```

#### 验证标准

各步骤的具体验收标准见 [五部职责定义](five-ministries-responsibilities.md)

**→ 进入第5阶段（所有Step通过后）**

---

### 第5阶段：集成验证（门下省Phase 3）

#### 最后的系统级验收

```
所有Step都通过Phase 2后:

门下省执行集成验证:
  ✓ 集成测试 100% 通过？
  ✓ E2E 测试 100% 通过？
  ✓ 需求功能点 100% 实现？
  ✓ 代码质量达标？
  ✓ 性能指标达标？
  ✓ 没有新增regression？

Decision:
  ✅ 全部通过 → 生成通过报告 → 交付皇帝
  ⚠️  有风险 → 生成条件通过报告 → 交付皇帝（标明风险）
  ❌ 失败 → 上报皇帝（详细说明问题）
```

**→ 进入第6阶段**

---

### 第6阶段：最终验收（皇帝）

#### 业务价值判定

```
皇帝接收报告，进行最终验收：
  ✓ 需求都达成了吗？
  ✓ 业务价值实现了吗？
  ✓ 质量可以接受吗？
  ✓ 可以发布上线吗？

Decision:
  ✅ 通过 → 业务验收完成
  ❌ 不通过 → 要求修改或重新规划
```

## 🎯 关键决策流程

### 当某个Step失败时

```
        Step失败
          ↓
     门下省验证
       /      \
    PASS      FAIL
      ↓         ↓
   继续    [第几次?]
         /          \
       1次            2次
       ↓              ↓
   让六部         上报皇帝
   重做            ↓
      ↓        皇帝决策
   重新验证      /    |    \
      ↓       继续  停止  重规划
     [Pass?]
     /     \
   YES    NO →继续重做或上报
```

### 当Step的输出不符合标准时

```
输出质量评估
  ↓
[能否调整标准?]
  ├─ YES → 调整标准
  │        └─ 继续下一步
  └─ NO → [能否让六部改进?]
           ├─ YES → 让六部重做
           └─ NO → 上报皇帝
```

## 🔄 工作流总结

```
【Phase 0】皇帝设定目标
   ↓ (目标 + domain.yaml)
【Phase 1】中书省规划
   ↓ (plan.json + dependencies)
【Phase 2】门下省审核计划
   ├─ REJECTED → 返回中书省
   │
   └─ APPROVED ↓
【Phase 3】尚书省执行（循环）
   ├─ for each step:
   │  ├─ 调用六部
   │  ├─ 报告给门下省
   │  └─ 等待门下省决策
   │
   ├─ Step失败 → 门下省决策
   │  ├─ 第1次失败 → 重做
   │  └─ 第2次失败 → 上报皇帝
   │
   └─ 所有Step通过 ↓
【Phase 4】门下省集成验证
   ├─ 运行系统级测试
   ├─ 验证需求覆盖
   └─ 生成最终报告 ↓
【Phase 5】皇帝最终验收
   ├─ 业务价值确认
   └─ 发布决策
```

## ⚠️ 常见陷阱和如何避免

### 陷阱1：中书省的计划太粗糙

```
问题：
  "实现支付功能" → 1个大Step
  门下省无法验证什么是"完成"

解决：
  拆成多个Step：
  ├─ Step 1: 分析现有支付系统
  ├─ Step 2: 设计新的支付接口
  ├─ Step 3: 实现核心逻辑
  ├─ Step 4: 添加错误处理
  └─ Step 5: 性能优化
```

### 陷阱2：验收标准太模糊

```
问题：
  "代码应该优雅"
  "性能应该好"
  → 无法定量评估

解决：
  {
    "code_quality": {
      "no_any_types": true,
      "coverage_min": 80,
      "complexity_max": 10
    },
    "performance": {
      "response_time_max_ms": 100,
      "memory_max_mb": 50
    }
  }
```

### 陷阱3：尚书省自行决策

```
问题：
  尚书省看到Step失败，自己决定："跳过这个，继续下一个"
  → 破坏流程，可能导致不一致

解决：
  尚书省只负责：
  ├─ 执行Step
  ├─ 报告结果
  └─ 等待门下省决策

  所有决策由门下省做
```

### 陷阱4：过度验证

```
问题：
  每个小Step都要求100%测试覆盖
  性能没有5%的下降余地
  → 执行不可能完成

解决：
  分类验证标准：
  ├─ 关键模块：100%覆盖、0%性能下降
  ├─ 普通模块：80%覆盖、5%性能下降
  └─ 非关键：70%覆盖、10%性能下降
```

### 陷阱5：六部越界

```
问题：
  工部（gongbu）在修改代码时，也运行了test
  → 破坏职责分离

解决：
  工部职责：只修改代码
  兵部职责：运行测试和验证

  清晰的工具权限控制：
  ├─ gongbu: edit, write（无bash）
  └─ bingbu: bash(readonly), read
```

## Q&A

### Q: 为什么需要三省而不是直接调用六部？
**A**:
- 中书省：负责规划，确保方向正确
- 门下省：负责质量把关，防止问题传播
- 尚书省：负责执行协调，确保按时完成
- 三层制约确保没有单点失败

### Q: 为什么要有两次验证机会？
**A**:
- 第1次失败可能是环境问题、参数问题等可修复的
- 允许一次重做，提高成功率
- 但防止无限循环，二次失败就上报
- 平衡自动修复和人工决策

### Q: 能否跳过某个Step？
**A**:
- 原则上不能跳过
- 但如果某个Step失败，经过评估确实影响不大
- 门下省可以决定跳过
- 但要记录决定和理由，供皇帝最终审查

### Q: 如何处理计划有bug的情况？
**A**:
- 执行过程中发现问题，停止执行
- 回到中书省，修改计划
- 门下省重新审核
- 这是最安全的做法，确保一致性

### Q: 为什么六部不能自己决策？
**A**:
- 六部是专业执行者，不是决策者
- 给予太多权力会导致不一致
- 通过三省的多层把关，确保质量
