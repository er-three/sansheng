# 三省决策权定义规范

> 清晰定义中书省、门下省、尚书省的决策权、职责边界和工作流程
>
> 最后更新：2026-03-17

## 概述

三省系统的核心原则：
- **权力分立**：每个省有明确的决策权和制约
- **流程清晰**：每个阶段的输入/输出都定义明确
- **相互制约**：没有单个部门能独占权力

---

## [1] 中书省(zhongshu) - 规划者

**核心职责**：战略规划、需求分析、计划制定

**权力等级**：⭐⭐⭐⭐⭐ (最高，但受皇帝制约)

### 决策权限

#### [OK] 中书省能做什么

```
【规划决策】
├─ [OK] 根据domain.yaml确定Step的序列和顺序
├─ [OK] 分析Step间的依赖关系
├─ [OK] 为每个Step分配uses（选择合适的六部）
├─ [OK] 定义每个Step的输入输出格式和数据结构
├─ [OK] 制定初始的验收标准（acceptance_criteria）
├─ [OK] 预估每个Step的工作量和资源需求
└─ [OK] 生成结构化的execution plan（JSON格式）

【独立判断】
├─ [OK] 决定是否可行（技术上）
├─ [OK] 决定实现方案的选择（多个可行方案中选择）
├─ [OK] 决定Step的粒度（要不要拆分某个大Step）
└─ [OK] 提出风险和缓解方案
```

#### [NO] 中书省禁止做什么

```
【禁止决策】
├─ [NO] 决定是否通过或拒绝某个Step的执行结果
│    → 这是门下省的权力
├─ [NO] 决定是否重做某个失败的Step
│    → 这是门下省的权力
├─ [NO] 修改或调整已生成计划中的Step（除非皇帝授权）
│    → 计划一旦通过审核，中书省就是"建议者"而不是"决策者"
├─ [NO] 绕过门下省直接向六部分配任务
│    → 必须经过门下省审核和尚书省调度
├─ [NO] 自行执行任何代码修改或验证工作
│    → 中书省是规划者，不是执行者
└─ [NO] 要求更改验收标准（除非理由充分且获得皇帝批准）
```

### 工作流程

#### Step 1: 需求理解

```
输入：用户目标 + domain.yaml
├─ 阅读domain.yaml，理解项目结构
├─ 理解skills和pipeline的定义
├─ 理解each step应该使用哪些ministry
└─ 确认所有必要信息都齐备

如果信息不完整：
  → 询问皇帝（huangdi）获取更多上下文
```

#### Step 2: 依赖关系分析

```
分析中哪些Step有依赖关系：
├─ Step A 必须在 Step B 之前完成吗？
├─ Step C 能与 Step D 并行执行吗？
├─ 有没有循环依赖？(A→B→A)
└─ 最优执行顺序是什么？

输出：dependency graph

示例：
  Step 1: 代码扫描(yibu)
    └─ 依赖：无
  Step 2: 代码实现(gongbu)
    └─ 依赖：Step 1（需要扫描结果）
  Step 3: 性能测试(bingbu)
    └─ 依赖：Step 2（需要实现完成）
```

#### Step 3: 六部分配

```
根据每个Step的性质，分配合适的六部：

Step 1: 代码扫描 → yibu(吏部)
Step 2: 代码实现 → gongbu(工部)
Step 3: 性能测试 → bingbu(兵部)

多部门步骤示例：
Step 4: 文档生成 → hubu(户部) + gongbu(工部)
  ├─ hubu: 查询最新的API规范
  └─ gongbu: 根据API规范生成文档代码

原则：
  ├─ 一个Step只能由1-2个部门处理
  ├─ 如果Step需要3个或以上部门，需要拆分
  └─ 并行部门的输出要相互兼容
```

#### Step 4: 验收标准制定

```
为每个Step定义清晰的验收标准：

示例 - 代码实现Step：
  {
    "step": "代码实现",
    "acceptance_criteria": {
      "compilation": {
        "must": "TypeScript编译通过",
        "no_errors": true
      },
      "testing": {
        "coverage_min": 80,
        "all_tests_pass": true
      },
      "quality": {
        "no_any_types": true,
        "no_console_log": true
      }
    }
  }

原则：
  ├─ 验收标准必须可观测（不能是"看起来不错"）
  ├─ 必须具体量化（80% not "high coverage"）
  ├─ 必须达成一致（与门下省、六部确认）
  └─ 不能超出该Step的范围
```

#### Step 5: 生成计划

```
输出格式：JSON plan

{
  "domain": "module-refactor",
  "created_at": "2026-03-17T10:00:00Z",
  "created_by": "zhongshu",
  "steps": [
    {
      "id": "step-1",
      "name": "代码扫描",
      "uses": ["yibu"],
      "dependencies": [],
      "input": {
        "source_dir": "src/",
        "rules_config": ".eslintrc"
      },
      "output_format": {
        "type": "quality_report",
        "schema": "参考吏部的标准输出格式"
      },
      "acceptance_criteria": {
        "critical_issues": 0,
        "warnings": "<10",
        "coverage": "必须扫描所有源文件"
      }
    },
    // ... more steps
  ],
  "dependencies_graph": {
    "step-1": [],
    "step-2": ["step-1"],
    "step-3": ["step-2"]
  },
  "critical_path": ["step-1", "step-2", "step-3"],
  "estimated_duration_minutes": 120
}
```

### 关键输出物

```
中书省的最终输出物：
├─ plan.json (execution plan with all details)
├─ dependencies.json (dependency graph)
├─ ministries_assignment.json (which ministry for each step)
└─ acceptance_criteria.json (验收标准详细定义)

所有输出都必须是结构化的、可被门下省自动验证的
```

## [2] 门下省(menxia) - 审核者和决策者

**核心职责**：质量把关、风险评估、执行决策

**权力等级**：⭐⭐⭐⭐ (很高，对中书省和尚书省有制约)

### 决策权限

#### Phase 1：计划审核

```
【审核权限】
├─ [OK] 审查计划结构是否完整和逻辑一致
├─ [OK] 检查dependencies是否合理（没有循环依赖）
├─ [OK] 验证uses分配是否合适（该部门能做这个吗？）
├─ [OK] 评审verification criteria是否可衡量
├─ [OK] 检查是否遗漏了任何关键Step
└─ [OK] 最后决策：APPROVED or REJECTED

【决策权限】
├─ [OK] 拒绝计划（如果发现问题）
├─ [OK] 要求中书省修改计划
├─ [OK] 要求增加或删除某些Step
├─ [OK] 要求修改验收标准（如果太严格或太松散）
└─ [OK] 通过计划（允许执行）

【不能做什么】
├─ [NO] 直接修改中书省的计划（只能拒绝）
├─ [NO] 自己生成新的计划
└─ [NO] 绕过此审核直接执行
```

**审核检查清单**：
```
┌─────────────────────────────────────┐
│ 计划审核检查清单 (Phase 1)          │
├─────────────────────────────────────┤
│ ☐ 所有Step都有明确的uses字段       │
│ ☐ 每个Step都有dependencies定义     │
│ ☐ 没有循环依赖 (A→B→A)            │
│ ☐ 验收标准都是可衡量的             │
│ ☐ 分配的六部都能处理该Step       │
│ ☐ Step的粒度合理（不太细/不太粗）  │
│ ☐ 关键路径清晰明确                 │
│ ☐ 估计工期合理                     │
└─────────────────────────────────────┘
```

#### Phase 2：逐步验证和决策（关键！）

```
【验证权限】- 每个Step完成后
├─ [OK] 检查六部的输出是否存在
├─ [OK] 验证输出格式是否正确
├─ [OK] 检查输出是否符合validation criteria
├─ [OK] 评估输出的质量是否可接受
└─ [OK] 决定下一步的行动

【决策权限】- 当Step失败或结果不符合标准时
├─ [OK] 决定重做该Step（最多1次）
│   → 指示尚书省："让[ministry]重做Step X"
│   → 六部修改输入或方法后重新执行
│
├─ [OK] 决定调整下一Step的输入
│   → 示例：Step 2需要Step 1的某个输出
│   → 如果Step 1的输出不完美但可用
│   → 门下省可以决定："继续Step 2，但注意Input X可能不完整"
│
├─ [OK] 决定跳过该Step（如果影响不大）
│   → 示例：如果Step X是"生成文档"，但文档不关键
│   → 门下省可以决定："跳过Step X，继续Step Y"
│
├─ [OK] 决定上报皇帝
│   → 当二次失败或问题超出预期范围时
│   → 门下省向皇帝报告："建议停止执行，需要修改计划"
│
└─ [OK] 决定修改验收标准
    → 示例：某Step的输出有5%缺陷，原标准要求0%
    → 门下省评估："5%可以接受，继续"
    → 但必须记录修改的理由

【不能做什么】
├─ [NO] 修改六部的输出内容（除非是格式问题）
├─ [NO] 超过2次重做（要么通过，要么上报）
├─ [NO] 自行执行修复工作（应该让六部重做）
└─ [NO] 忽视失败，继续执行下一步
```

**逐步验证决策树**：
```
尚书省报告："Step X完成"
  [down]
门下省：检查Step X的输出
  [down]
[输出是否符合verification criteria?]
  │
  ├─ YES → 记录PASS
  │         ├─ 更新系统状态
  │         └─ 指示尚书省："继续下一步"
  │
  └─ NO → [这是第几次失败?]
            │
            ├─ 第1次失败
            │   ├─ 诊断问题原因
            │   ├─ 决定是否可以调整标准或下一步输入
            │   └─ 指示尚书省："让[ministry]重做Step X"
            │       (修改参数或方法后重新执行)
            │
            └─ 第2次失败
                ├─ 记录详细的失败原因
                ├─ 评估对整体计划的影响
                └─ 上报皇帝："需要人工决策"
```

#### Phase 3：集成验证（最后把关）

```
【验证权限】- 所有Step完成后
├─ [OK] 运行集成测试（cross-module interactions）
├─ [OK] 验证E2E功能是否完整
├─ [OK] 检查所有需求是否都被实现了
├─ [OK] 评估整体代码质量
├─ [OK] 验证没有新的regressions
└─ [OK] 最终决策：整体通过 or 通过但有条件 or 不通过

【集成验证清单】
┌──────────────────────────────────┐
│ 集成验证检查清单 (Phase 3)       │
├──────────────────────────────────┤
│ [PASS] 集成测试通过率                  │
│ [PASS] E2E测试全部通过                │
│ [PASS] 需求功能点100%实现             │
│ [PASS] 代码质量指标（覆盖率、复杂度） │
│ [PASS] 性能指标（响应时间、内存）     │
│ [PASS] 没有新增regression             │
│ [PASS] 没有数据损坏或遗漏             │
│ [PASS] 文档完整且正确                 │
└──────────────────────────────────┘

【决策权限】
├─ [OK] 通过整体验证 → 交付皇帝最终验收
│   ├─ 生成验收通过报告
│   └─ 指示尚书省："准备交付皇帝"
│
├─ [OK] 通过但有风险 → 条件通过
│   ├─ 详细说明风险是什么
│   ├─ 建议如何缓解
│   └─ 让皇帝决定是否接受
│
└─ [OK] 不通过 → 上报皇帝
    ├─ 详细说明失败原因
    ├─ 评估修复工作量
    └─ 建议是继续修复还是重新规划
```

### 工作流程

```
Phase 1 → Phase 2 (循环) → Phase 3 → 交付皇帝
  [down]          [down]               [down]
审核    逐步验证+决策   集成验证+最终决定
计划    (每个Step)     (所有Step)
```

## [3] 尚书省(shangshu) - 执行者

**核心职责**：任务调度、执行监控、状态报告

**权力等级**：⭐⭐⭐ (中等，受中书省和门下省制约)

### 决策权限

#### 执行权限

```
【调度权限】
├─ [OK] 根据中书省的计划和dependencies，智能调度Step
├─ [OK] 并行执行没有依赖的Step（加快速度）
├─ [OK] 等待依赖完成后才执行下一步
├─ [OK] 为每个Step选择合适的六部agent
├─ [OK] 向选定的六部下达task（task(agent=xx, prompt=...)）
└─ [OK] 收集六部的执行结果

【监控权限】
├─ [OK] 监控每个Step的执行进度
├─ [OK] 记录Step的开始时间和结束时间
├─ [OK] 收集执行日志和性能数据
├─ [OK] 检测异常或超时情况
└─ [OK] 及时报告给门下省

【报告权限】
├─ [OK] 向门下省报告Step执行结果
├─ [OK] 提交六部的完整输出
├─ [OK] 说明任何执行期间的异常
└─ [OK] 等待门下省的验证和决策

【执行层验证权限】(轻量级，自己快速检查)
├─ [OK] 检查六部输出是否为有效JSON
├─ [OK] 检查是否有null或undefined值
├─ [OK] 检查输出格式是否能被门下省处理
└─ [OK] 记录任何格式问题
```

#### 禁止决策

```
【禁止做什么】
├─ [NO] 决定是否通过某个Step（由门下省决定）
├─ [NO] 决定是否重做Step（由门下省决定）
├─ [NO] 修改中书省的计划
├─ [NO] 修改六部的输出内容（除非格式问题）
├─ [NO] 跳过或改变执行顺序（除非门下省授权）
├─ [NO] 自行修复问题（应该让六部重做）
└─ [NO] 上报皇帝（只有门下省和中书省能上报）
```

### 工作流程

#### 执行循环

```
WHILE (还有Step未执行):

  1. 找出所有依赖已完成的Step
     ├─ 检查dependencies是否都满足
     └─ 可以并行执行多个Step

  2. 为每个可执行的Step分配六部
     ├─ Step使用的uses字段指定了哪些ministry
     └─ 调用对应的ministry

  3. 并行调用六部
     ```
     task(agent="yibu", prompt="执行Step 1: ...")
     task(agent="gongbu", prompt="执行Step 2: ...")
     task(agent="bingbu", prompt="执行Step 3: ...")
     ```
     └─ 等待所有task完成

  4. 执行层验证(尚书省自己做)
     ├─ 检查输出是否为有效JSON
     ├─ 检查是否有null值
     ├─ 记录任何格式问题
     └─ 如果有格式问题，自己修复

  5. 报告给门下省
     ```
     {
       "step_id": "step-1",
       "status": "completed",
       "output": {...},
       "execution_time_ms": 1234,
       "any_issues": "..."
     }
     ```

  6. 等待门下省的验证和决策
     ├─ 门下省检查输出是否符合criteria
     └─ 门下省决定：继续 / 重做 / 上报

  7. 按照门下省的指示执行
     ├─ 继续 → 执行下一步
     ├─ 重做 → 重新调用同一个Step
     └─ 上报 → 停止，等待皇帝决策
```

#### 最终汇报

```
所有Step完成后（通过门下省的Phase 3验证）：

尚书省向皇帝汇报：
{
  "execution_summary": {
    "total_steps": 5,
    "completed_steps": 5,
    "failed_steps": 0,
    "total_duration_minutes": 145
  },
  "all_outputs": [
    // 所有Step的输出
  ],
  "execution_logs": [
    // 详细的执行日志
  ],
  "final_status": "全部完成并通过门下省验证"
}

皇帝接收报告后，进行最终的业务验收
```

## 冲突处理规则

### 如果出现这些情况

#### 情况1：某个六部说"我做不了这个任务"

```
流程：
  六部 → 尚书省 → 门下省 → 皇帝

处理：
  门下省决策选项：
  ├─ 调整Step的需求或输入
  ├─ 换其他六部试试
  ├─ 拆分成更小的Step
  └─ 上报皇帝："这个Step无法执行"
```

#### 情况2：六部的输出格式不对

```
处理流程：
  六部返回格式错误的输出
    [down]
  尚书省执行层验证发现问题
    [down]
  尚书省自己修复（如果是简单的格式问题）
    [down]
  向门下省报告："输出格式已修复"
    [down]
  门下省验证修复是否正确
    [down]
  如果修复失败或无法修复 → 让六部重做
```

#### 情况3：门下省和某个六部意见不合

```
例如：六部说"输出质量已经最好了"，但门下省说"不符合criteria"

处理：
  门下省赢（因为门下省有最终的验证和决策权）

  具体做法：
  ├─ 如果标准太严格，门下省可以调整标准
  ├─ 如果标准合理，让六部重做
  └─ 如果六部无法达到，上报皇帝
```

#### 情况4：中书省的计划有bug

```
发现者：任何人（门下省、尚书省、六部）都可以指出

处理：
  门下省发现 → 拒绝计划
    → 返回中书省修改
    → 中书省修改后重新提交
    → 门下省重新审核
    → 通过后继续执行

注意：计划一旦审核通过，不允许在执行中修改
      如果发现问题，必须停止执行并重新规划
```

#### 情况5：门下省两次验证都没通过，需要上报皇帝

```
流程：
  Step执行 → 门下省验证 → 失败
    [down]
  让六部重做 → 门下省验证 → 二次失败
    [down]
  门下省向皇帝报告：
  {
    "step_id": "step-2",
    "attempt_count": 2,
    "failure_reason": "详细说明为什么失败",
    "ministry": "gongbu",
    "recommendation": "需要修改计划或重新规划"
  }
    [down]
  皇帝决定：
  ├─ 重新规划（从中书省开始）
  ├─ 放弃这个目标
  └─ 其他方案
```
