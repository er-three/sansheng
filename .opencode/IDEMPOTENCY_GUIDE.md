# 六部执行幂等性指南

**目的**: 确保 Agent 重试执行不会产生副作用或数据重复

**原则**: 每个六部 Agent 在重新执行同一步骤时，不应该产生不同的结果或污染现有数据。

---

## 什么是幂等性（Idempotency）

```
f(x) = f(f(x))  // 多次执行结果相同
```

在本系统中：
- **第一次执行** 步骤 X：生成输出 O
- **重试** 步骤 X（失败后）：应该生成相同的 O，而不是 O'
- **不应该**：覆盖、追加、或遗漏数据

---

## 各部幂等性保证

### 吏部（yibu）- 文件读取和审查

**操作类型**: 纯读取，天然幂等 [OK]

**实现**:
- 不修改文件
- 每次执行相同的检查逻辑
- 输出相同的审查结果

**重试安全**: [OK] 完全安全
- 读取同一文件 N 次，结果不变

**示例**:
```
第 1 次：检查 module.ts → 生成报告 report-001
重  试：检查 module.ts → 生成相同的 report-001
```

---

### 工部（gongbu）- 代码生成和修改

**操作类型**: 写入/修改文件，需要特殊处理

**实现**:
1. **检查是否存在** — 执行前检查输出文件是否已存在
2. **如果存在** — 对比哈希值，确保内容一致
3. **如果不存在** — 执行生成操作
4. **验证输出** — 生成后验证输出文件哈希值

**重试安全**:
- [OK] 如果输出文件存在且内容一致 → 直接返回成功
- [OK] 如果输出文件不存在 → 生成新文件
- [WARN]  如果输出文件内容不一致 → 拒绝执行，上报尚书省

**示例**:
```
第 1 次执行：写 output.ts → hash-abc123
重  试：检查 output.ts 存在且 hash=abc123 [PASS]
        → 返回成功（不重新生成）

第 1 次执行：写 output.ts → hash-abc123
重  试：检查 output.ts 存在但 hash=xyz789 ✗
        → 拒绝执行，上报冲突
```

**Hash 验证实现** (伪代码):
```typescript
const outputPath = "src/generated/output.ts"
const expectedContent = generateCode(...)
const expectedHash = md5(expectedContent)

// 检查是否存在
if (fileExists(outputPath)) {
  const actualHash = md5(readFile(outputPath))
  if (actualHash === expectedHash) {
    return { status: "success", note: "已存在且内容一致" }
  } else {
    return { status: "failure", error: "输出文件已存在但内容不匹配" }
  }
}

// 不存在，执行写入
writeFile(outputPath, expectedContent)
return { status: "success" }
```

### 户部（hubu）- 外部资源查询

**操作类型**: 纯查询，天然幂等 [OK]

**实现**:
- webfetch/websearch 是无状态的
- 同一个 URL 查询 N 次得到相同或相似的结果
- 外部状态变化（如网页更新）是 API 侧的变化，不是 Agent 的问题

**重试安全**: [OK] 完全安全
- 网络超时 → 重试查询相同的 URL
- API 限流 → 重试后可能成功
- 结果略有差异（新闻时间戳等）→ 正常

**示例**:
```
第 1 次：查询 API 端点 X → 获得结果 R1
重  试：查询 API 端点 X → 获得结果 R1（或略微更新的 R1'）
```

---

### 礼部（libu）- 技能工具调用

**操作类型**: 取决于技能实现

**实现**:
- 大多数 skill 应该是幂等的（参见 domain.yaml 中的 skill 定义）
- 如果 skill 不是幂等的，需要在 skill 定义中明确标记
- 礼部调用前应该检查 skill 的幂等性标记

**重试安全**: [WARN] 依赖 skill 实现

**标记方式** (在 domain.yaml 中):
```yaml
skills:
  generate_scaffold:
    idempotent: true    # 可以安全重试
    check_exists: true  # 检查输出是否已存在

  deploy_service:
    idempotent: false   # 禁止重试（有副作用）
    note: "部署操作不幂等，失败需要手动干预"
```

### 兵部（bingbu）- 测试执行

**操作类型**: 查询（测试结果），天然幂等 [OK]

**实现**:
- 执行测试命令，收集输出
- 相同的命令多次执行，结果应该相同（或至少测试结果相同）
- 不应该修改测试代码或修改被测试的代码

**重试安全**: [OK] 完全安全
- 测试超时 → 重试同一测试命令
- 网络中断（API 调用） → 重试后通常成功
- 结果应该稳定

**示例**:
```
第 1 次：npm test → 3 pass, 1 fail
重  试：npm test → 3 pass, 1 fail（同一个 test 仍然失败）
```

---

### 刑部（xingbu）- 代码审查

**操作类型**: 纯读取 + 分析，天然幂等 [OK]

**实现**:
- 不修改代码，只做审查
- 相同的代码审查 N 次，结果应该相同（或至少问题列表相同）
- 如果代码在两次审查间被修改，那是被审查对象的变化，不是审查本身的问题

**重试安全**: [OK] 完全安全
- 审查超时 → 重试同一代码审查
- 网络中断 → 重试后继续

**示例**:
```
第 1 次：审查 module.ts → 发现 5 个问题
重  试：审查 module.ts → 发现相同的 5 个问题
```

---

### 库部（kubu）- 文档和规范编辑

**操作类型**: 写入文档，需要检查

**实现**:
1. **生成文档内容** 时，基于规范的版本号或时间戳
2. **检查是否存在** — 执行前检查输出文件
3. **如果存在** — 对比版本，确保内容一致
4. **如果不存在** — 写入新文档

**重试安全**:
- [OK] 文档内容确定性（基于规范版本）→ 重试安全
- [WARN]  如果文档内容不确定 → 需要显式版本控制

**示例**:
```
第 1 次：生成 API.md（基于规范 v1.2） → hash-abc123
重  试：检查 API.md 存在且 hash=abc123 [PASS]
        → 返回成功（不重新生成）

版本更新后：
重  试：生成 API.md（基于规范 v1.3） → hash-xyz789 ≠ abc123
        → 拒绝执行，要求人工审批版本更新
```

## 重试场景应对表

| 场景 | 原六部 | 操作 | 结果 |
|------|--------|------|------|
| 网络超时 | yibu (read) | 重试查询相同文件 | [OK] 安全 |
| 网络超时 | gongbu (write) | 检查输出文件 hash，重试 | [OK] 安全 |
| 网络超时 | hubu (search) | 重试查询同一 URL | [OK] 安全 |
| 验收失败 | 任意 | 重做同一步骤 | [OK]/[WARN]（见上表） |
| 输出文件冲突 | gongbu | 中止，上报尚书省 | [OK] 安全 |
| 文件内容变化 | yibu | 检测变化，上报 | [OK] 安全 |

## 尚书省重试协议

尚书省在调用六部重试时，**必须**：

1. **保持输入不变** — 传入完全相同的参数
2. **允许检查输出** — 不强制删除已生成的输出
3. **检查冲突** — 如果输出不一致，拒绝继续

尚书省**禁止**：
- 修改 skill 参数试图"绕过"失败
- 清除输出文件试图强制重新生成
- 假设重试一定成功（要有失败回退）

---

## 验收（御史台）的幂等性视角

御史台在验收相同的执行结果时，应该给出**相同的验收结论**。

这意味着：
- 如果 Step 1 第一次验收通过 → 重复验收应该仍然通过
- 如果 Step 1 第一次验收失败 → 重复验收应该仍然失败（除非执行结果实际改变）

```
验收逻辑：verify_step(step_id, execution_result)
          [down]
          判断 execution_result 的输出是否满足检查项
          [down]
          输出 PASS / FAIL（相同输入 → 相同输出）
```
