# 三省六部制工作流程详解

> 以"提取 asset-manager 资产"为例，完整追踪每一层的信息流。

---

## 整体分工

```
用户输入
    ↓
皇帝（huangdi）   决策层：切换领域、设变量、启动三省、最终验收
    ↓
Plugin            基础设施：自动注入约束、自动验收、自动压缩状态
    ↓
中书省（zhongshu） 规划层：制定步骤、确定顺序和依赖
    ↓
门下省（menxia）   审核层：否决计划、验收每步结果
    ↓
尚书省（shangshu） 执行层：按序调度六部 Skill
    ↓
六部（Skills）     技能层：专业执行各司其职
    ↓
皇帝（huangdi）   业务验收：目标是否真正达成
```

---

## 第一步：皇帝接收任务

**触发时机**：用户说"提取 asset-manager 资产"

**皇帝做的事**：
1. 调用 `list_domains()` 了解可用领域
2. 判断任务类型 → 调用 `switch_domain("asset-management")`
   - Plugin 执行：更新 `registry.json`
   - Plugin 执行：重建 `.opencode/skills/` 软链接
3. 调用 `set_variables({ module_name: "asset-manager" })`
4. 向用户确认目标和成功标准

**皇帝关心**：
- ✅ 目标是什么
- ✅ 用哪个领域
- ❌ 不管 Phase 怎么执行
- ❌ 不管验收命令是什么

---

## 第二步：Plugin 自动注入（用户不感知）

**触发时机**：每次 LLM 调用前，`experimental.chat.system.transform` 执行

**自动注入到系统提示词**：

```
## 当前领域：asset-management
描述：旧代码资产提取，并行提取+强制验证

### 全局约束
- 禁止压缩、禁止模糊词（等/…/相关逻辑）
- 必须行号绑定
- 禁止框架代码混入
- 生成后立即落盘返回

### 流水线状态
⬜ 吏部（scan）：代码扫描
⬜ 兵部（extract）：并行资产提取
⬜ 礼部（mapping）：UI框架映射
⬜ 刑部（behavior）：行为场景提取
⬜ 工部（detect）：框架代码检测
⬜ 户部（verify）：一致性核验

### 当前变量
- module_name: asset-manager
```

**效果**：
- 所有 Agent 每次对话都能看到约束和状态
- 不需要在每个 task 指令里手动重复传入
- **原来 6 次重复传入 → 现在 0 次**

---

## 第三步：中书省规划

**触发时机**：皇帝说 `@zhongshu 请为 asset-manager 制定提取计划`

**中书省做的事**：
1. 读取已注入的流水线状态（不需要额外调用）
2. 检查相关目录和文件是否存在
3. 输出结构化计划：

```json
{
  "domain": "asset-management",
  "variables": { "module_name": "asset-manager" },
  "steps": [
    { "slot": "scan",     "input": "src/asset-manager/",         "depends_on": [] },
    { "slot": "extract",  "input": "code-index.yaml",            "depends_on": ["scan"] },
    { "slot": "mapping",  "input": "ui.yaml + 原始HTML",         "depends_on": ["extract"] },
    { "slot": "behavior", "input": "logic.ts + 源码",            "depends_on": ["extract"] },
    { "slot": "detect",   "input": "ui.yaml + logic.ts",         "depends_on": ["mapping","behavior"] },
    { "slot": "verify",   "input": "所有7个资产文件",             "depends_on": ["detect"] }
  ]
}
```

**中书省关心**：
- ✅ 步骤是什么、顺序和依赖
- ❌ 不管验收命令是什么
- ❌ 不管工具怎么调用

---

## 第四步：门下省审核计划

**触发时机**：皇帝说 `@menxia 请审核上述计划`

**门下省检查**：

| 检查项 | 结果 |
|--------|------|
| 顺序是否符合 pipeline？ | ✅ |
| extract 依赖 scan，scan 在前？ | ✅ |
| detect 依赖 mapping 和 behavior，两者在前？ | ✅ |
| module_name 变量已设置？ | ✅ |
| 源码目录存在？ | → 询问用户确认 |

**两种结果**：
- **通过** → "计划审核通过，可以执行"
- **驳回** → "第3步 mapping 的输入文件 ui.yaml 还不存在，请等 extract 完成后再列入" → 打回中书省重新规划

**门下省有真实的否决权，不是走过场。**

---

## 第五步：尚书省执行

**触发时机**：皇帝说 `@shangshu 计划已审核通过，请按序执行`

**尚书省做的事**：

```
Step 1 → 加载 scan Skill → 调用 opencoder-7
  task(agent="opencoder-7", skill="scan", prompt="执行代码扫描")
  ← 完成
  → 汇报门下省验收

Step 2 → 加载 extract Skill（发现是并行步骤）
  在一个回复中同时发出：
    task(agent="opencoder-7",  skill="extract-ui",    ...)
    task(agent="opencoder-8",  skill="extract-logic",  ...)
    task(agent="opencoder-9",  skill="extract-infra",  ...)
    task(agent="opencoder-10", skill="extract-arch",   ...)
  ← 4个 Agent 并行执行
  → 汇报门下省验收

... 以此类推
```

**尚书省关心**：
- ✅ 严格按序执行
- ✅ 失败立即停止上报
- ❌ 不做任何判断和规划

---

## 第六步：Plugin 自动验收（门下省的执行手）

**触发时机**：每次 Skill 调用完成后，`tool.execute.after` 自动触发

**以 scan 步骤为例**，Plugin 读取 `contract.yaml`：

```yaml
verify:
  - type: file_exists
    path: "specs/asset-manager/code-index.yaml"
    error_msg: "code-index.yaml 不存在"
```

**执行验收**：
- 文件存在 → 在输出追加：
  ```
  ✅ 门下省验收通过：代码扫描
  进度：1/6 步完成
  下一步：兵部（extract）
  ```
  同时更新 `registry.json` 的 `pipeline_state`

**以 extract 步骤为例**，检查5个文件：
- 全部存在 → 通过，进度 2/6
- 任意缺失 → 在输出追加：
  ```
  ❌ 门下省验收失败：并行资产提取
  - logic.ts 不存在
  请修复后重新执行此步骤，不得进入下一步
  ```
  尚书省看到后立即停止，上报皇帝

---

## 失败处理流程

```
某步 Skill 执行失败
    ↓
Plugin 验收：检测到文件不存在 / 命令失败
    ↓
输出中追加失败信息
    ↓
尚书省：看到失败，停止，不进入下一步，上报皇帝
    ↓
皇帝：决策
    ├── 重试 → 让尚书省重新执行该步骤
    ├── 修改策略 → 让中书省重新规划
    └── 人工介入 → 告知用户需要手动处理
```

---

## Context 压缩时的状态保留

**触发时机**：Session 接近 context 上限，`experimental.session.compacting` 触发

**Plugin 自动保留**：
```
当前领域：asset-management
已完成步骤：scan, extract, mapping
当前步骤：behavior
当前变量：{ "module_name": "asset-manager" }
以上是三省六部制流水线的关键状态，必须保留。
```

**效果**：压缩后 AI 依然知道从哪里继续，不会重头开始。

---

## 最终：皇帝业务验收

**触发时机**：户部（verify）完成，Plugin 验收通过

**皇帝从业务角度验收**：
- 6条引用链全部闭合了吗？
- design-spec.md 内容完整吗？
- 可以进入逆向工程阶段了吗？

**两种后续**：
- 验收通过 → 告知用户完成
- 直接切换 → `/switch-domain reverse-engineering` → 同一 Session 继续下一阶段

---

## 各层职责边界总结

| 角色 | 关心什么 | 不做什么 |
|------|---------|---------|
| **皇帝** | 商业目标、战略方向、最终验收 | 技术细节、代码实现 |
| **Plugin** | 注入约束、验收执行、状态管理 | 业务逻辑判断 |
| **中书省** | 任务规划、步骤顺序和依赖 | 执行、验收 |
| **门下省** | 标准把关、质量验收、否决权 | 规划、执行 |
| **尚书省** | 按序执行、驱动六部 | 规划、判断、验收 |
| **六部** | 专业技能实现 | 规划、决策 |

---

## 与原版单文件的关键差异

| 对比项 | 原版 asset-manager.md | 三省六部制 |
|--------|----------------------|-----------|
| 全局约束 | 每次 task 调用手动传入（×6） | Plugin 自动注入（×0） |
| 验收逻辑 | 分散在各 Phase 里 | contract.yaml 集中定义 |
| 失败处理 | 写在 Orchestrator 里 | Plugin + 尚书省自动处理 |
| 执行框架 | 每个场景重复一遍 | AGENTS.md 写一次共用 |
| 场景切换 | 关闭 session 重开 | `/switch-domain` 同 session |
| 状态压缩 | 无 | Plugin 自动保留关键状态 |
| 维护成本 | 改约束需改 12 处 | 改约束只改 domain.yaml |
