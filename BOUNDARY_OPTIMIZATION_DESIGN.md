# 三省六部制 Agent 边界优化设计

**分析日期**: 2026-03-16
**状态**: 待实施

---

## 当前边界问题

### 问题 1：门下省双重职责

```
现状：
门下省 = 计划审核（前置）+ 执行验收（后置）

问题：
- 两个阶段输入完全不同
  - 前置：接收中书省的 JSON 计划
  - 后置：接收六部的执行结果，调用 verify_step
- 同一 agent 在流程里出现两次，职责混淆
- steps: 10 对两个职责都偏少
```

### 问题 2：中书省的隐性依赖

```
现状：
中书省需要知道 domain.yaml 的 pipeline 顺序来规划步骤
但 permission: edit=deny, bash=deny，且没有 read 权限声明

问题：
- 它怎么知道 pipeline 顺序？靠皇帝传入？靠记忆？
- 隐性依赖，边界不透明
- 一旦 domain 变更，中书省的规划可能悄悄出错
```

### 问题 3：失败回路断链

```
现状：
验收失败 → 尚书省停止 → 上报皇帝 → ？

问题：
- 皇帝收到失败报告后做什么？重新规划？直接让六部重做？
- 没有明确的修复路径
- 皇帝被迫介入技术细节，违反"只做战略"的原则
```

### 问题 4：礼部与尚书省职责重叠（轻微）

```
现状：
- 尚书省：task() 调度六部 agent
- 礼部：skill() 调度技能工具

在实际 pipeline 里，是尚书省传 skill 参数给六部 agent
礼部什么时候被调用不够清晰
```

## 优化设计

### 核心改动：门下省一分为二

```
现在：
中书省 → 门下省（审计划）→ 尚书省 → 六部 → 门下省（验收）

优化后：
中书省 → 门下省（审计划）→ 尚书省 → 六部 → 御史台（验收）
```

**门下省**：职责收窄为纯计划审核
- 输入：中书省的 JSON 执行计划
- 输出：`APPROVED` / `REJECTED + issues[]`
- steps: 10 足够

**御史台（新增）**：纯执行结果验收
- 输入：步骤 ID + 六部执行结果
- 输出：`PASS` / `FAIL + failed_checks[]`
- 调用 `verify_step`，不做其他事
- model: haiku 即可（纯机械验收）

---

### 中书省：补充 read 权限 + 显式读取 domain

```
优化：
1. 添加 read/glob 权限（只读）
2. 工作流第一步改为：
   读取 .opencode/domains/{domain}/domain.yaml
   提取 pipeline 顺序
   再基于 pipeline 制定计划
3. 输出计划时附上 pipeline_source 字段，证明来源可追溯
```

---

### 尚书省：补充一次重试 + 明确修复路径

```
优化：
验收失败时：
  第 1 次失败 → 通知对应六部重做（不上报皇帝）
  第 2 次失败 → 上报皇帝

修复路径明确为：
  御史台 FAIL → 尚书省通知原六部重做 → 御史台再验收
  两次失败 → 皇帝裁决
```

## 优化后结构图

```
用户目标
   ↓
皇帝（战略确认 + 最终验收）
   ↓
中书省（读 domain.yaml → 规划 JSON 计划）
   ↓
门下省（审计划：顺序/输入/变量）
   ↓
尚书省（调度执行）
   ├── 步骤内并行 → 六部执行
   │                    ↓
   │               御史台（verify_step）
   │                    ↓
   │          PASS → 下一步 / FAIL → 六部重做(1次) → 再验收
   │                              二次失败 → 皇帝
   └── 全部完成 → 皇帝最终验收
```

## 边界验证标准

每个 Agent 满足以下三条才算边界合格：

1. **单一职责** — 只回答一个问题
2. **输出自闭合** — 输出本身是完整的，不需要上下文才能理解
3. **可独立测试** — 给固定输入，结果可预期
